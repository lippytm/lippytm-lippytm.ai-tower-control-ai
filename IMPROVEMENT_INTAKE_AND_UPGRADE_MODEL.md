# Improvement Intake and Upgrade Model

This document defines how the wider `lippytm` ecosystem can capture, review, prioritize, and apply improvements across learning systems, workflow systems, product systems, sandbox systems, and community systems.

## Purpose
Keep continuous improvement organized as the ecosystem grows in depth, detail, diversity, flexibility, and specialization.

## Core Principle
A good ecosystem should make it easy to notice improvement opportunities, record them clearly, review them in small loops, and convert the strongest ones into reusable upgrades.

## Improvement Sources

### 1. Builder Feedback
Use for:
- lesson confusion
- setup friction
- missing examples
- repeated questions

### 2. Workflow Feedback
Use for:
- unclear workflow steps
- weak review checkpoints
- poor sandbox separation
- support friction

### 3. Product Feedback
Use for:
- unclear kit packaging
- weak support notes
- missing update guidance
- variant confusion

### 4. Community Feedback
Use for:
- contributor friction
- member pathway confusion
- partner support gaps
- collaboration improvement ideas

## Upgrade Flow
1. Record the improvement clearly
2. Link it to the affected layer or repo
3. Review whether it is local or reusable
4. Improve the specific system
5. Promote repeated improvements into templates, kits, or standards

## Best Practices
- keep improvement intake simple
- separate local fixes from reusable upgrades
- attach security and review notes when the improvement affects risk boundaries
- use repeated confusion as a signal for stronger templates or lessons
- archive obsolete patterns when better upgrades replace them
