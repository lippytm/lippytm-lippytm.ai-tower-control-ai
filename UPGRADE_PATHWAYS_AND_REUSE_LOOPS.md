# Upgrade Pathways and Reuse Loops

This document defines how the wider `lippytm` ecosystem can turn improvements, lessons learned, support updates, and specialization work into reusable upgrades across repositories.

## Purpose
Make ecosystem growth more repeatable by giving every strong improvement a path toward reuse instead of leaving it as a one-off fix.

## Core Principle
A good improvement should move through a loop: identify, clarify, improve, review, reuse, and either promote or retire.

## Upgrade Pathways

### 1. Local Upgrade Path
Use for:
- repo-specific fixes
- setup-note improvements
- local workflow clarifications
- small support updates

### 2. Reusable Upgrade Path
Use for:
- repeated fixes across repos
- repeated setup friction
- repeated lesson confusion
- repeated workflow improvements

### 3. Template Upgrade Path
Use for:
- improvements that should become starter patterns
- improvements that should change kit structures
- stronger defaults for future variants

### 4. Standard Upgrade Path
Use for:
- improvements that should change control-tower guidance
- repeated security clarifications
- stronger review expectations
- clearer packaging or maintenance rules

## Reuse Loop
1. notice the weak point
2. document the improvement clearly
3. apply the local fix
4. review whether the same issue appears elsewhere
5. promote repeated fixes into templates, kits, or standards
6. archive weaker older patterns when better upgrades replace them

## Best Practices
- keep upgrade notes simple enough to repeat
- separate local fixes from system-level upgrades
- connect repeated improvement paths back to the control tower
- attach security and review notes when the upgrade affects risk boundaries
- prefer reusable loops over isolated patches when the same confusion keeps returning
