# Specialization Review and Refresh Guide

This document defines how specialization tracks should be reviewed and refreshed across the wider `lippytm` ecosystem.

## Purpose
Help specialized tracks stay useful, teachable, and maintainable as the ecosystem grows into deeper programming, machine learning, blockchain, and workflow areas.

## Core Principle
A specialization track should be reviewed in small loops so it can improve, simplify, or retire weak paths before confusion multiplies.

## Review Areas

### 1. Setup Clarity
Review:
- setup notes
- dependency notes
- environment assumptions
- troubleshooting guidance

### 2. Learning Clarity
Review:
- lesson usefulness
- exercise quality
- example clarity
- review-question usefulness

### 3. Security Clarity
Review:
- safer defaults
- sandbox boundaries
- scope notes
- review checkpoints

### 4. Reuse Value
Review:
- whether the track should become a pack
- whether repeated steps should become templates
- whether weaker branches should be archived

## Best Practices
- keep reviews small enough to repeat often
- connect refreshes back to strong base systems
- archive weak specialization paths clearly
- promote durable patterns into reusable packs or templates
- keep setup, review, and security notes attached to specialization work
