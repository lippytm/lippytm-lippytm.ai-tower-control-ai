# System Variant Governance

This document defines how reusable variants of repositories, templates, kits, and educational systems should stay organized across the wider `lippytm` ecosystem.

## Purpose
Keep replicated systems maintainable as the ecosystem grows across memberships, affiliate programs, products, content systems, and specialized learning environments.

## Core Principle
A variant should be free to adapt for its own audience or purpose, while still keeping enough shared structure to remain teachable, reviewable, and maintainable.

## Variant Types

### 1. Learning Variants
Use for:
- beginner versions
- advanced versions
- topic-specific lesson branches
- industry-specific education tracks

### 2. Product Variants
Use for:
- starter kits
- premium versions
- support bundles
- service delivery variants

### 3. Distribution Variants
Use for:
- membership tiers
- affiliate versions
- partner versions
- audience-specific content systems

### 4. Sandbox Variants
Use for:
- experimental branches
- safe practice spaces
- research adaptations
- review and reflection spaces

## Shared Requirements
Every strong variant should keep:
- a clear purpose
- setup notes
- review notes
- security notes
- customization boundaries
- links back to shared standards when useful

## Best Practices
- keep the base stable
- document what changed in the variant
- separate experimental variants from operational variants
- keep security and review habits with each derivative system
- promote mature variants into reusable templates when they prove useful
