# Sandbox to Application Promotion Guide

## Purpose
This guide defines how sandbox outputs, simulation findings, and experimental system results can be promoted into stronger applications, products, services, or reusable factory assets across the lippytm ecosystem.

## Core Goal
Create a clear path from:
- sandbox experiment
- to scenario packet
- to diagnostic findings
- to reusable pattern
- to packaged application or solution
- to wider portfolio deployment

## Promotion Stages
### Stage 1 — Experimental
The object remains inside a sandbox and is still exploratory.

### Stage 2 — Structured
The sandbox result has clearer inputs, outputs, and documented observations.

### Stage 3 — Repeatable
The result can be run again with similar conditions and produce useful learning.

### Stage 4 — Packaged
The result can be turned into a template, profile, packet, component, or application package.

### Stage 5 — Promoted
The result becomes part of a broader product, service, application, or factory asset.

## Promotion Questions
For each sandbox output ask:
- what was learned?
- what repeated well?
- what still fails or needs work?
- what object type could this become next?
- what repo, product, or platform should receive it?
- what review level applies after promotion?

## Suggested Promotion Targets
- application component
- AI solution package
- workflow template
- learning module
- diagnostics packet
- CRM support asset
- media or narrative support asset
- business-growth application

## Best Practices
- do not promote one-off results without repeatability evidence
- keep sandbox history linked to promoted assets
- separate experimental notes from packaged outputs
- use scorecards and diagnostics to support promotion decisions
- prefer promotions that strengthen more than one system where appropriate
