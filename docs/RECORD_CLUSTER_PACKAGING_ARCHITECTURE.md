# Record Cluster Packaging Architecture

## Purpose
This document defines how live record clusters should be packaged so they remain usable, linked, and promotion-ready across the lippytm ecosystem.

## Core Goal
Turn groups of live records into clear operating packages rather than letting records accumulate without structure.

## Packaging Layers
### Layer 1 — Core Records
Quality, implementation, improvement, and repo-health records.

### Layer 2 — Support Records
Builder records, sandbox records, scorecards, lineage records, repeatability records.

### Layer 3 — Promotion Records
Bundle records, package records, AI solution records, application records, growth records.

### Layer 4 — Tracker Records
Coverage files, remediation files, cluster-level files, health files.

## Packaging Rule
A scope becomes easier to operate when its records can be read as one cluster package instead of many scattered files.

## Best Practices
- build core records first
- add support records second
- add promotion records when the cluster is stable enough
- update cluster trackers whenever a scope gains a new record type
- use cluster packages to choose the next best scope for promotion

## Best Current Need
The strongest next packaging work is to deepen the narrative clusters and add sandbox/repeatability support to the strongest repo clusters.
