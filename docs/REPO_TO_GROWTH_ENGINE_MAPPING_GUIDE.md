# Repo to Growth Engine Mapping Guide

## Purpose
This guide defines how to map repositories to repeatable growth engines across the lippytm ecosystem.

## Core Questions
For each repository ask:
- what source assets does this repo contain?
- what growth engines could those assets feed?
- what products, services, applications, or bundles can grow from it?
- what educational, media, CRM, or reporting surfaces could strengthen it?
- what related repos can reuse the same engine?
- what review level applies to the delivery surfaces?

## Suggested Mapping Fields
- repo ID
- repo category
- linked growth engine IDs
- linked product IDs
- linked service IDs
- linked application IDs
- linked platform IDs
- linked concept IDs
- linked component IDs
- maturity stage
- priority wave
- notes

## Suggested Engine Categories
- direct product engine
- service packaging engine
- application engine
- educational-business engine
- media/campaign engine
- CRM workflow engine
- reporting leverage engine
- cross-domain reuse engine

## Best Practices
- map highest-value repos first
- let one repo feed more than one growth engine when appropriate
- connect every engine mapping back to stable IDs and trackers
- distinguish internal enablement value from public delivery value
- review repo-to-engine mappings after major product, service, or architecture changes
