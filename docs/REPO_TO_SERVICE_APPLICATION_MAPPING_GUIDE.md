# Repo to Service Application Mapping Guide

## Purpose
This guide defines how to map repositories to services, offers, products, and applications across the lippytm Business of Businesses ecosystem.

The goal is to make it easier to see how a repo can support not only internal architecture, but also outward-facing service and application opportunities.

## Core Mapping Questions
For each repository ask:
- what services can this repo enable?
- what applications can this repo support?
- what offers or packages can be created from it?
- what educational or media outputs can be repurposed from it?
- what platforms can deliver those outputs?
- what review level applies to those delivery surfaces?

## Suggested Mapping Fields
- repo ID
- repo category
- linked product IDs
- linked service IDs
- linked application IDs
- linked platform IDs
- linked cost center IDs
- linked blueprint IDs
- linked profile IDs
- maturity stage
- priority wave
- notes

## Suggested Service Types
- consulting-style service
- implementation service
- educational service
- CRM / workflow service
- reporting / accountability service
- media/content service
- bot deployment service
- integration service

## Suggested Application Types
- internal ops application
- business-growth application
- customer-facing application
- educational application
- media/campaign application
- reporting application
- knowledge/reference application

## Best Practices
- let one repo map to more than one service or application when appropriate
- keep service/application mapping explicit in profiles and catalogs
- separate internal-only surfaces from public-facing surfaces
- connect service/application mappings back to product and inventory objects
