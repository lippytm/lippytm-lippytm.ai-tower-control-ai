# Live Record Cluster Quality Model

## Purpose
This document defines how to assess the quality of live operating record clusters across the lippytm ecosystem.

## Core Goal
Move from isolated live records toward stronger record clusters that give each important scope enough operational density to support real work.

## What a Record Cluster Means
A record cluster is a group of linked live records around one scope, such as:
- quality packet
- implementation packet
- improvement loop packet
- repo health profile
- builder record
- sandbox record
- scorecard or repeatability record

## Cluster Quality Levels
### Level 1 — Starter Cluster
At least one live record exists for the scope.

### Level 2 — Working Cluster
At least two to three linked live records exist and support a real operating path.

### Level 3 — Managed Cluster
Quality, implementation, improvement, and health records exist together.

### Level 4 — Promotion-Ready Cluster
The scope also includes builder/sandbox lineage or repeatability support.

### Level 5 — High-Density Cluster
The scope has a strong live set across records, scorecards, lineage, and promotion paths.

## Quality Dimensions
- scope clarity
- linkage quality
- operational usefulness
- promotion readiness
- remediation support
- reuse value

## Best Practices
- build clusters in waves instead of creating scattered isolated records
- prioritize high-value scopes first
- treat missing cluster parts as remediation targets
- use cluster quality to choose the next best record to create
