# Replication Lifecycle and Maintenance

This document defines how replicated systems should be maintained across the wider `lippytm` ecosystem.

## Purpose
Keep cloned, forked, templated, and variant systems useful over time instead of letting them drift into confusion or decay.

## Core Principle
A reusable system should have a visible lifecycle: base creation, replication, customization, review, maintenance, and either promotion, archive, or retirement.

## Lifecycle Stages

### 1. Base Stage
Use for:
- stable starting systems
- shared standards
- reusable templates
- common setup and review patterns

### 2. Replication Stage
Use for:
- clones
- forks
- template-based starts
- audience-specific copies

### 3. Customization Stage
Use for:
- local adaptation
- feature additions
- audience-specific content
- membership or affiliate variants

### 4. Review Stage
Use for:
- security review
- documentation review
- setup review
- support review
- product/readiness review

### 5. Maintenance Stage
Use for:
- updates
- dependency refreshes
- support-note revisions
- security-note revisions
- content refresh cycles

### 6. Outcome Stage
Possible outcomes:
- promote into a reusable template
- keep as an active maintained variant
- archive as reference
- retire when no longer useful

## Best Practices
- keep lifecycle status visible in docs
- review replicated systems in small loops
- document when a variant diverges significantly from the base
- keep maintenance notes close to setup notes
- archive stale systems instead of letting them confuse active users
