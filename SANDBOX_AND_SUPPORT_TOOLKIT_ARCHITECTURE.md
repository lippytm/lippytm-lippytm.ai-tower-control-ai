# Sandbox and Support Toolkit Architecture

This document defines how sandbox environments and support toolkits should be organized across the wider `lippytm` AI, computer science, machine learning, blockchain, and educational ecosystem.

## Purpose
Create a modular architecture where each learning path can have its own support toolkit and sandbox structure without losing shared standards, security habits, or cross-repo coordination.

## Core Principle
Every major builder path should have:
- a learning pathway
- a support toolkit pattern
- a sandbox pattern
- a review pattern
- a reuse path into templates, kits, or educational applications

## Toolkit Layers

### 1. Language Support Toolkits
Examples:
- Python support notes
- JavaScript / TypeScript support notes
- shell and command-line support notes
- SQL support notes
- optional system-language support as the ecosystem grows

Focus:
- setup guidance
- library guidance
- debugging guidance
- reusable examples

### 2. Machine Learning Support Toolkits
Examples:
- model API usage notes
- evaluation notes
- retrieval and embeddings patterns
- workflow experimentation support
- review and safety notes

Focus:
- practical experimentation
- structured review
- reusable ML learning patterns

### 3. Blockchain Support Toolkits
Examples:
- wallet and account guidance
- smart contract workflow guidance
- testing and deployment notes
- chain interaction support patterns
- safe experimentation notes

Focus:
- applied project learning
- clear boundaries
- reusable builder structures

### 4. Environment Support Toolkits
Examples:
- Linux-first setup notes
- mixed-environment setup notes
- dependency and secret-handling notes
- environment hygiene checklists
- sandbox setup guidance

Focus:
- reproducibility
- portability
- operational clarity

## Sandbox Layers

### 1. Lesson Sandboxes
Use for:
- small exercises
- walkthroughs
- isolated examples
- repeated teaching loops

### 2. Workflow Sandboxes
Use for:
- agent experiments
- automation patterns
- routing and evaluation loops
- secure testing before broader reuse

### 3. Blockchain Sandboxes
Use for:
- contract experiments
- wallet practice
- testing flows
- isolated builder learning

### 4. Reflection Sandboxes
Use for:
- review loops
- lessons learned logs
- architecture notes
- self-improvement and skill review

## Best Practices
- keep each sandbox clearly scoped
- separate production systems from experimental systems
- include security notes in every toolkit and sandbox
- keep support toolkits tied to real builder tasks
- convert mature sandbox patterns into reusable educational kits
- link every major toolkit path back to the control tower
